# Migrating from v0.4 to v0.5 — Odoo Semantic MCP

> **Audience:** Existing API key holders whose AI tool config (Claude Code, Cursor, Codex, Gemini, Custom GPT, VS Code, Antigravity) was set up against v0.4.x. v0.5 is **fully backward-compatible** — your snippets keep working — but a few legacy tools now emit a `DEPRECATED` banner and will be removed in v0.6.

## TL;DR

- Tool surface grew from **21** to **28** tools, plus **7 new MCP Resources** (`odoo://` URI scheme).
- Three new **supersets** consolidate 10 legacy tools behind a single `method=` discriminator (ADR-0028).
- Four new **session-context** tools let you pin an Odoo version once per chat session (ADR-0029).
- The 10 legacy tools they supersede still answer in v0.5 — they just emit a `DEPRECATED:` banner. **They are removed in v0.6.**
- Plan to swap callers (your AI client snippets, your prompts) before the v0.6 release.

## What Changed

### New supersets (preferred — M11 Wave D)

| New tool | Discriminator (`method=`) | Replaces |
|----------|---------------------------|----------|
| `model_inspect` | `fields` / `methods` / `views` / `all` | `resolve_model`, `list_fields`, `list_methods`, `list_views` |
| `module_inspect` | `describe` / `fields` / `views` / `owl` / `qweb` / `patches` | `describe_module` (still kept), `list_views` (module-scoped), `list_owl_components`, `list_qweb_templates`, `list_js_patches` |
| `entity_lookup` | `kind=field` / `kind=method` / `kind=view` | `resolve_field`, `resolve_method`, `resolve_view` |

### New session-context tools (M11 Wave E)

| Tool | Purpose |
|------|---------|
| `set_active_version(odoo_version)` | Pin the Odoo version for this session (24h TTL per API key). |
| `set_active_profile(profile_name)` | Pin the tenant profile for multi-tenant MCP deployments. |
| `list_available_versions()` | Discover which Odoo versions the server has indexed. |
| `list_available_profiles()` | Discover which tenant profiles exist. |

### New MCP Resources (M11 Wave F, ADR-0030)

Read-only URI-addressable handles — preferred when you already know the entity ID and just want the canonical record without a tool call:

```
odoo://17.0/model/sale.order
odoo://17.0/field/sale.order/amount_total
odoo://17.0/method/sale.order/action_confirm
odoo://17.0/module/sale_management
odoo://17.0/view/sale.view_order_form
odoo://17.0/pattern/computed-monetary-aggregate
odoo://17.0/stylesheet/website_sale/static/src/scss/website_sale.scss
```

Same `X-API-Key` header as tool calls.

## Side-by-Side Migration Examples

### Example 1 — Inspect a model fully

```diff
- # v0.4 — three separate calls:
- resolve_model(model_name="sale.order", odoo_version="17.0")
- list_fields(model="sale.order", odoo_version="17.0")
- list_methods(model="sale.order", odoo_version="17.0")

+ # v0.5 — one call after pinning the version:
+ set_active_version("17.0")                                  # once per session
+ model_inspect(model="sale.order", method="all")             # consolidated tree
```

### Example 2 — Look up a single field

```diff
- # v0.4:
- resolve_field(model_name="sale.order", field_name="amount_total", odoo_version="17.0")

+ # v0.5:
+ entity_lookup(kind="field", model="sale.order", field="amount_total")
```

### Example 3 — Module UI inventory

```diff
- # v0.4 — three calls:
- list_owl_components(module="website_sale", odoo_version="17.0")
- list_qweb_templates(module="website_sale", odoo_version="17.0")
- list_js_patches(odoo_version="17.0", module="website_sale")

+ # v0.5 — one call (per artefact kind) or one describe call:
+ module_inspect(module="website_sale", method="owl")
+ module_inspect(module="website_sale", method="qweb")
+ module_inspect(module="website_sale", method="patches")
```

### Example 4 — Look up a view

```diff
- resolve_view(view_id="sale.view_order_form", odoo_version="17.0")
+ entity_lookup(kind="view", xmlid="sale.view_order_form")
```

## What You Gain

- **Fewer tool-list scrolling** — AI clients display 28 tools vs. an inflated list; the three supersets cover the common cases up front.
- **Shared schemas via discriminator** — `method=`/`kind=` route requests within one tool, reducing prompt-window noise and avoiding "which list_* do I call" cognitive overhead (ADR-0028).
- **Opaque ref drill-down** — `model_inspect` / `module_inspect` responses include opaque refs you can hand straight to `entity_lookup` without re-parsing names, plus `odoo://` URIs you can bookmark.
- **Session context** — pin once, omit `odoo_version=` everywhere afterwards. Stops the cargo-cult repetition of the version string on every call (ADR-0029).
- **MCP Resources** — bookmark-stable read access. URIs are stable across server restarts and indexer re-runs as long as the entity exists.

## What Stays the Same

- All v0.4 tools still respond — `DEPRECATED:` banner only, output structure unchanged.
- HTTP transport, `X-API-Key` header, and JSON-RPC envelope are identical.
- Tools not listed in the supersession table (`find_examples`, `impact_analysis`, `lookup_core_api`, `api_version_diff`, `find_deprecated_usage`, `lint_check`, `cli_help`, `suggest_pattern`, `check_module_exists`, `find_override_point`, `describe_module`) are **unchanged**.
- Persona behaviour, ranking heuristic (ADR-0013), tree grammar (ADR-0023), and authorization model are all stable.

| Version | Status of legacy tools | Action |
