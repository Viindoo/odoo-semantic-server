## 3. Fix 1: P1-D — Add HSTS Header to Nginx

**Root cause:** Nginx `server {}` block thiếu `add_header Strict-Transport-Security` directive.

**Verification symptom:**
```bash
curl -sI https://odoo-semantic.viindoo.com/health | grep -i strict-transport
# → (empty — KHÔNG có header)
```

### Diagnose

```bash
# Tìm nginx config block:
sudo grep -n "Strict-Transport\|add_header" /etc/nginx/sites-available/odoo-semantic*
# → nếu empty: directive chưa có → proceed với fix bên dưới
```

### Fix

Mở nginx config:

```bash
sudo nano /etc/nginx/sites-available/odoo-semantic
```

Thêm dòng sau **bên trong** block `server { ... }` cho port 443 (sau `server_name` hoặc `ssl_certificate` lines):

```nginx
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
```

Ví dụ vị trí chính xác:

```nginx
server {
    listen 443 ssl;
    server_name odoo-semantic.viindoo.com;
    ssl_certificate     /etc/letsencrypt/live/odoo-semantic.viindoo.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/odoo-semantic.viindoo.com/privkey.pem;

    # Thêm dòng này:
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;

    # ... các location block ...
}
```

### Apply + Verify

```bash
# Test config trước khi reload:
sudo nginx -t
# → nginx: configuration file /etc/nginx/nginx.conf syntax is OK
# → nginx: configuration file /etc/nginx/nginx.conf test is successful

# Reload (không downtime):
sudo systemctl reload nginx

# Verify:
curl -sI https://odoo-semantic.viindoo.com/health | grep -i strict-transport
# → Strict-Transport-Security: max-age=31536000; includeSubDomains
```

### Rollback nếu cần

```bash
sudo cp ~/backups/m7.5-hotfix/nginx-odoo-semantic.bak \
    /etc/nginx/sites-available/odoo-semantic
sudo systemctl reload nginx
```

**Estimated time:** 5 phút.

## 4. Fix 2: P1-A — Ollama SSL CERTIFICATE_VERIFY_FAILED

**Root cause:** `odoo-semantic.conf` section `[embedder]` có `url = https://localhost:11434` thay vì `http://`. Ollama không có TLS certificate hợp lệ → Python `requests` raise `CERTIFICATE_VERIFY_FAILED`.

**Affected tools:** `find_examples` (§6 tool #5) và `suggest_pattern` (§6 tool #12) — bất cứ tool nào gọi embedder.

### Diagnose

```bash
# Đọc cấu hình hiện tại:
sudo grep -A5 '\[embedder\]' /etc/odoo-semantic/odoo-semantic.conf
# Expected BAD value: url = https://localhost:11434
# Expected GOOD value: url = http://localhost:11434
```

Nếu thấy `https://` → proceed fix. Nếu thấy `http://` mà tool vẫn fail → xem §4.1 bên dưới.

### Fix

```bash
sudo nano /etc/odoo-semantic/odoo-semantic.conf
```

Tìm section `[embedder]`, đổi:

```ini
# TRƯỚC (SAI):
[embedder]
url = https://localhost:11434

# SAU (ĐÚNG):
[embedder]
url = http://localhost:11434
```

Lưu file, sau đó restart MCP:

```bash
sudo systemctl restart odoo-semantic-mcp
sudo systemctl status odoo-semantic-mcp
# → Active: active (running)
```

### Verify

Từ server — xác nhận Ollama reachable:

```bash
curl http://localhost:11434/api/tags
# → {"models":[...]} — list models available
```

Từ client (Claude Code với plugin configured) — xác nhận embedding tools hoạt động:

```
find_examples("compute tax based on partner country")
# Expected: 5 results với file path + similarity score

suggest_pattern("computed field cross-model partner_id")
# Expected: 3-5 PatternExample với code snippet + gotchas
```

### §4.1 Alternative: Ollama trên remote host với TLS

Nếu Ollama chạy trên remote host và cần TLS (không dùng `localhost`):

```bash
# Option A: Dùng http:// với SSH tunnel thay vì TLS public endpoint
# (xem docs/deploy/embedder-setup.md §4 cho SSH tunnel setup)

# Option B: Nếu cần TLS thật — add CA cert vào trusted store (out of scope hotfix)
# Liên hệ admin infra để cấu hình cert properly
```

### Rollback nếu cần

```bash
sudo cp ~/backups/m7.5-hotfix/odoo-semantic.conf.bak \
    /etc/odoo-semantic/odoo-semantic.conf
sudo systemctl restart odoo-semantic-mcp
```

**Estimated time:** 5 phút.

## 5. Fix 3+4: P1-B + P1-C — Re-index CoreSymbol + CLICommand

**Root cause (bundled):** Production core index chưa bao gồm đủ symbols từ Odoo 17.0 (và 16.0). Hai triệu chứng:
- P1-B: `lookup_core_api("name_get", "17.0")` → trống (CoreSymbol không có `name_get`)
- P1-C: `cli_help("server", "--gevent-port", "17.0")` → trống (CLICommand không có `--gevent-port`)

Cả hai fix bằng **cùng một command** — `index-core` index toàn bộ CoreSymbol + CLICommand + LintRule cho một version.

### Prerequisites

```bash
# Verify git repos có trên server:
ls ~/git/odoo17/odoo-bin 2>/dev/null && echo "odoo17 OK" || echo "odoo17 MISSING"
ls ~/git/odoo16/odoo-bin 2>/dev/null && echo "odoo16 OK" || echo "odoo16 MISSING"
```

Nếu repo chưa có:

```bash
# Clone odoo17 (nếu cần):
git clone --branch 17.0 --depth=1 https://github.com/odoo/odoo.git ~/git/odoo17

# Clone odoo16 (nếu cần):
git clone --branch 16.0 --depth=1 https://github.com/odoo/odoo.git ~/git/odoo16
```

Đảm bảo mirror up to date:

```bash
cd ~/git/odoo17 && git pull --ff-only && cd -
cd ~/git/odoo16 && git pull --ff-only && cd -
```

### Pull Latest Code

```bash
cd /opt/odoo-semantic-mcp
git pull --ff-only
# (hoặc branch chứa PR #82 nếu chưa merge)
```

Activate venv:

```bash
source ~/.venv/odoo-semantic-mcp/bin/activate
```

### Run index-core cho Odoo 17.0

```bash
ODOO_SEMANTIC_CONF=/etc/odoo-semantic/odoo-semantic.conf \
python -m src.indexer index-core \
    --odoo-root ~/git/odoo17 \
    --version 17.0

# Expected output (cuối run):
# ✓ CoreSymbol: <N> symbols indexed for 17.0
# ✓ CLICommand: <M> commands, <K> flags indexed for 17.0
```

Có thể chạy background nếu muốn tiếp tục làm Fix 5 song song:

```bash
nohup sh -c 'ODOO_SEMANTIC_CONF=/etc/odoo-semantic/odoo-semantic.conf \
    python -m src.indexer index-core \
    --odoo-root ~/git/odoo17 --version 17.0' \
    > /tmp/index-core-17.log 2>&1 &

echo "PID: $! — tail /tmp/index-core-17.log để theo dõi"
```

### Run index-core cho Odoo 16.0

```bash
ODOO_SEMANTIC_CONF=/etc/odoo-semantic/odoo-semantic.conf \
python -m src.indexer index-core \
    --odoo-root ~/git/odoo16 \
    --version 16.0
```

### Verify

Từ client (Claude Code với plugin):

```
lookup_core_api("name_get", "17.0")
# Expected: symbol returned — description có nội dung, version_added hoặc version_removed present
# NOTE: status sẽ là 'stable' (xem §9 Known Quirks bên dưới — đây là đúng)

api_version_diff("name_get", "16.0", "17.0")
# Expected: diff entry — thấy thay đổi hoặc "no breaking change" message

cli_help("server", "--gevent-port", "17.0")
# Expected: flag description — "Specify the port the server is listening to for gevent"
```

**Estimated time:** 15–30 phút tùy theo kích thước repo và tốc độ disk.

```bash
# → nếu "active (running)": skip sang A3 (nginx upstream issue)
```

**Step A3 — Kiểm tra nginx upstream:**

```bash
sudo grep -A5 'location.*admin\|location /admin' \
    /etc/nginx/sites-available/odoo-semantic*
# Expected: proxy_pass http://127.0.0.1:8003
```

Nếu thiếu `location /admin` block — thêm vào nginx config:

```nginx
location /admin/ {
    proxy_pass http://127.0.0.1:8003/;
    proxy_set_header Host $host;
    proxy_set_header X-Real-IP $remote_addr;
    proxy_set_header X-Forwarded-For $proxy_add_x_forwarded_for;
    proxy_set_header X-Forwarded-Proto $scheme;
}
```

Apply:

```bash

```

**Verify Branch A:**

```bash

# Expected: HTTP/2 302 Location: /admin/login (nếu chưa đăng nhập)
```

## 8. Known Quirks (Post-Fix)

### P2 — `name_get` Shows `status='stable'` After Re-Index

**Behavior:** Sau khi `index-core` hoàn tất, `lookup_core_api("name_get", "17.0")` trả về symbol với `status='stable'` — không phải `status='deprecated'`.

**Lý do:** Odoo 17 dùng **runtime `DeprecationWarning`** (phát ra lúc chạy) thay vì decorator `@api.deprecated` (static annotation) cho `name_get`. Parser của `index-core` scan static code — không detect được runtime warning.

**Đây là đúng behavior của parser hiện tại** — không phải bug trong fix này. Tool trả về symbol với đầy đủ thông tin, chỉ `status` thiếu độ chính xác.

**Track as:** P2 enhancement — cần parser aware của runtime deprecation patterns (vd: detect `warnings.warn(... DeprecationWarning)` trong source). Tạo issue trong TASKS.md nếu muốn prioritize.

**Impact:** `find_deprecated_usage("17.0")` sẽ **không** list `name_get` là deprecated usage — đây là false negative. AI clients cần biết quirk này khi tư vấn về `name_get` deprecation.

```bash
sudo nginx -t && sudo systemctl reload nginx

# Verify rollback:
```

### Rollback Fix 2 (Ollama URL)

```bash

    /etc/odoo-semantic/odoo-semantic.conf
sudo systemctl restart odoo-semantic-mcp

```

### Rollback Fix 3+4 (Re-index)

Re-index không destructive — thêm/update CoreSymbol nodes trong Neo4j, không xóa existing data. Nếu muốn revert về state trước:

```bash
# Option A: Restore Neo4j từ snapshot (xem docs/deploy/disaster-recovery.md §2):
docker compose stop neo4j
docker cp ~/backups/m7.5-hotfix/neo4j-<TIMESTAMP>.dump \
    odoo-semantic-mcp-neo4j-1:/data/backups/
docker compose run --rm neo4j \
    neo4j-admin database load neo4j \
    --from-path=/data/backups \
    --overwrite-destination=true
docker compose start neo4j

# Option B: Nếu snapshot không có — index lại từ data cũ (không có cách undo đơn giản).
# Re-index không phá data existing, chỉ thêm/update — không cần rollback logic cho Fix 3+4.
```

### Rollback Fix 5 (Web UI)

```bash
# Nếu started service và muốn stop lại:
sudo systemctl stop odoo-semantic-webui

# Nếu sửa nginx và muốn revert:
sudo cp ~/backups/m7.5-hotfix/nginx-odoo-semantic.bak \
    /etc/nginx/sites-available/odoo-semantic
sudo nginx -t && sudo systemctl reload nginx
```

### Khi Nào Rollback

- Smoke test: `curl http://127.0.0.1:8002/health` → không trả `{"neo4j": "ok", "postgres": "ok"}`
- Service `odoo-semantic-mcp` fail restart sau 3 lần (`systemctl status` → `failed`)
- Tool client gọi bất kỳ tool nào return error (không phải empty result) sau fix
