## 3. Fix 1: P1-D — Add HSTS Header to Nginx

**Root cause:** Nginx `server {}` block thiếu `add_header Strict-Transport-Security` directive.

**Verification symptom:**
```bash
curl -sI https://odoo-semantic.viindoo.com/health | grep -i strict-transport
# → (empty — KHÔNG có header)
```

### Diagnose

```bash
# Tìm nginx config block:
sudo grep -n "Strict-Transport\|add_header" /etc/nginx/sites-available/odoo-semantic-mcp*
# → nếu empty: directive chưa có → proceed với fix bên dưới
```

### Fix

Mở nginx config:

```bash
sudo nano /etc/nginx/sites-available/odoo-semantic-mcp
```

Thêm dòng sau **bên trong** block `server { ... }` cho port 443 (sau `server_name` hoặc `ssl_certificate` lines):

```nginx
add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;
```

Ví dụ vị trí chính xác:

```nginx
server {
    listen 443 ssl;
    server_name odoo-semantic.viindoo.com;
    ssl_certificate     /etc/letsencrypt/live/odoo-semantic.viindoo.com/fullchain.pem;
    ssl_certificate_key /etc/letsencrypt/live/odoo-semantic.viindoo.com/privkey.pem;

    # Thêm dòng này:
    add_header Strict-Transport-Security "max-age=31536000; includeSubDomains" always;

    # ... các location block ...
}
```

### Apply + Verify

```bash
# Test config trước khi reload:
sudo nginx -t
# → nginx: configuration file /etc/nginx/nginx.conf syntax is OK
# → nginx: configuration file /etc/nginx/nginx.conf test is successful

# Reload (không downtime):
sudo systemctl reload nginx

# Verify:
curl -sI https://odoo-semantic.viindoo.com/health | grep -i strict-transport
# → Strict-Transport-Security: max-age=31536000; includeSubDomains
```

### Rollback nếu cần

```bash
sudo cp ~/backups/m7.5-hotfix/nginx-odoo-semantic.bak \
    /etc/nginx/sites-available/odoo-semantic-mcp
sudo systemctl reload nginx
```

**Estimated time:** 5 phút.

## 4. Fix 2: P1-A — Embedder SSL/Connectivity Failure

> **Actual prod root cause (2026-05-14):** Remote embedder URL có **wrong port `:9999`** (port đã closed/firewalled — connection timeout, không phải TLS issue). Original section dưới đây (§4.1) mô tả Ollama localhost TLS failure mode — vẫn valid cho deployments dùng local Ollama, nhưng KHÔNG match prod. Xem **§4.2** cho fix thực tế đã áp dụng.

**Affected tools:** `find_examples` (§6 tool #5) và `suggest_pattern` (§6 tool #12) — bất cứ tool nào gọi embedder.

### §4.1 Original failure mode: Ollama localhost với `https://`

**Root cause (template deployment):** `odoo-semantic.conf` section `[embedder]` có `url = https://localhost:11434` thay vì `http://`. Ollama không có TLS certificate hợp lệ → Python `requests` raise `CERTIFICATE_VERIFY_FAILED`.

#### Diagnose

```bash
# Đọc cấu hình hiện tại:
sudo grep -A5 '\[embedder\]' /etc/odoo-semantic/odoo-semantic.conf
# Expected BAD value: url = https://localhost:11434
# Expected GOOD value: url = http://localhost:11434
```

Nếu thấy `https://` localhost → proceed §4.1 fix. Nếu URL là **remote host với port lạ** (vd `:9999`) → bỏ qua, sang **§4.2**.

#### Fix

```bash
sudo nano /etc/odoo-semantic/odoo-semantic.conf
```

Tìm section `[embedder]`, đổi:

```ini
# TRƯỚC (SAI):
[embedder]
url = https://localhost:11434

# SAU (ĐÚNG):
[embedder]
url = http://localhost:11434
```

Lưu file, sau đó restart MCP:

```bash
sudo systemctl restart odoo-semantic-mcp
sudo systemctl status odoo-semantic-mcp
# → Active: active (running)
```

#### Verify

Từ server — xác nhận Ollama reachable:

```bash
curl http://localhost:11434/api/tags
# → {"models":[...]} — list models available
```

Từ client (Claude Code với plugin configured) — xác nhận embedding tools hoạt động:

```
find_examples("compute tax based on partner country")
# Expected: 5 results với file path + similarity score

suggest_pattern("computed field cross-model partner_id")
# Expected: 3-5 PatternExample với code snippet + gotchas
```

#### Alternative: Ollama trên remote host với TLS

Nếu Ollama chạy trên remote host và cần TLS (không dùng `localhost`):

```bash
# Option A: Dùng http:// với SSH tunnel thay vì TLS public endpoint
# (xem docs/deploy/embedder-setup.md §4 cho SSH tunnel setup)

# Option B: Nếu cần TLS thật — add CA cert vào trusted store (out of scope hotfix)
# Liên hệ admin infra để cấu hình cert properly
```

#### Rollback nếu cần

```bash
sudo cp ~/backups/m7.5-hotfix/odoo-semantic.conf.bak \
    /etc/odoo-semantic/odoo-semantic.conf
sudo systemctl restart odoo-semantic-mcp
```

**Estimated time §4.1:** 5 phút.

### §4.2 Actual prod fix: Remote Embedder Wrong Port

**Áp dụng cho:** Deployments dùng **remote embedder** (vd `https://embed.viindoo.com`, không phải Ollama localhost). M7.5 verification report ban đầu chẩn đoán nhầm là §4.1; sau khi điều tra deeper, prod thật sự bị **wrong port** trong URL, không phải TLS issue.

**Root cause:** `odoo-semantic.conf` `[embedder] url = https://embed.viindoo.com:9999`. Port `9999` đã closed/firewalled tại embed gateway. Nginx serve embed endpoint qua **port 443 mặc định**. Code `src/indexer/embedder.py:88` append `/api/embed` vào base URL → POST tới `https://embed.viindoo.com:9999/api/embed` → connection timeout → MCP tool fail.

#### Diagnose

```bash
# 1. Đọc conf hiện tại (path conf tùy deployment — có thể là ~/git/odoo-semantic-mcp/odoo-semantic.conf hoặc /etc/odoo-semantic/odoo-semantic.conf):
grep -A5 '\[embedder\]' ~/git/odoo-semantic-mcp/odoo-semantic.conf
# Nếu thấy url với port lạ (không phải 443/11434) → proceed.

# 2. Network probe — confirm port nào reachable:
curl -sI -m 5 https://embed.viindoo.com:9999/    # → timeout (port closed)
curl -sI -m 5 https://embed.viindoo.com/         # → HTTP/2 404 (nginx OK)
curl -sI -m 5 https://embed.viindoo.com/api/embed
# → HTTP/2 401 (endpoint reachable, requires Bearer token = đúng behavior)
```

#### Fix (1 dòng)

```bash
# Conf file editable không cần sudo nếu thuộc home user (vd ~/git/...):
nano ~/git/odoo-semantic-mcp/odoo-semantic.conf

# Đổi line 19 (hoặc dòng tương ứng):
# TRƯỚC (SAI):  url = https://embed.viindoo.com:9999
# SAU (ĐÚNG):   url = https://embed.viindoo.com
```

Restart MCP (sudo):

```bash
sudo systemctl restart odoo-semantic-mcp
sudo systemctl status odoo-semantic-mcp
# → Active: active (running)
```

#### Verify

```bash
# Server-side reachability:
curl -sI -m 5 https://embed.viindoo.com/api/embed
# → HTTP/2 401 (auth required without token = expected)

# Client-side via MCP tool:
#   find_examples("compute tax based on partner country") → 5 results với similarity score
#   suggest_pattern("computed field cross-model partner_id") → 3-5 PatternExample
```

#### Rollback

```bash
cp ~/backups/m7.5-hotfix/odoo-semantic.conf.bak \
   ~/git/odoo-semantic-mcp/odoo-semantic.conf
sudo systemctl restart odoo-semantic-mcp
```

**KHÔNG cần** — sau khi confirm fix work:
- ❌ Code change (zero hardcode `9999` trong codebase — Sonnet audit 2026-05-14)
- ❌ Re-embed PatternExample (same server, same model `qwen3-embedding-q5km`, same `dim=1024`)
- ❌ CA cert config (server có valid TLS cert tại port 443)

**Estimated time §4.2:** 3 phút.

## 5. Fix 3+4: P1-B + P1-C — Core Index Backfill

**Root cause (bundled):** Production core index chưa bao gồm đủ symbols. Hai triệu chứng:
- P1-B: `lookup_core_api("name_get", "17.0")` → trống (CoreSymbol cho v17 = 0)

Cả hai fix bằng **cùng một command** — `index-core` populate CoreSymbol + CLICommand + CLIFlag + LintRule cho một version.

> **Runbook correction (2026-05-14):** CLI subcommand thực tế là `--source` (NOT `--odoo-root`). Path examples nên match prod convention `~/git/odoo_<V>.0` thay vì `~/git/odoo<V>`. Sections dưới đã được sửa.

## 8. Known Quirks (Post-Fix)

### P2 — `name_get` Shows `status='stable'` After Re-Index

**Behavior:** Sau khi `index-core` hoàn tất, `lookup_core_api("name_get", "17.0")` trả về symbol với `status='stable'` — không phải `status='deprecated'`.

**Lý do:** Odoo 17 dùng **runtime `DeprecationWarning`** (phát ra lúc chạy) thay vì decorator `@api.deprecated` (static annotation) cho `name_get`. Parser của `index-core` scan static code — không detect được runtime warning.

**Đây là đúng behavior của parser hiện tại** — không phải bug trong fix này. Tool trả về symbol với đầy đủ thông tin, chỉ `status` thiếu độ chính xác.

**Track as:** P2 enhancement — cần parser aware của runtime deprecation patterns (vd: detect `warnings.warn(... DeprecationWarning)` trong source). Tạo issue trong TASKS.md nếu muốn prioritize.

**Impact:** `find_deprecated_usage("17.0")` sẽ **không** list `name_get` là deprecated usage — đây là false negative. AI clients cần biết quirk này khi tư vấn về `name_get` deprecation.


### Rollback Fix 3+4 (Re-index)

Re-index không destructive — thêm/update CoreSymbol nodes trong Neo4j, không xóa existing data. Nếu muốn revert về state trước:

```bash
# Option A: Restore Neo4j từ snapshot (xem docs/deploy/disaster-recovery.md §2):
docker compose stop neo4j
docker cp ~/backups/m7.5-hotfix/neo4j-<TIMESTAMP>.dump \
    odoo-semantic-mcp-neo4j-1:/data/backups/
docker compose run --rm neo4j \
    neo4j-admin database load neo4j \
    --from-path=/data/backups \
    --overwrite-destination=true
docker compose start neo4j

# Option B: Nếu snapshot không có — index lại từ data cũ (không có cách undo đơn giản).
# Re-index không phá data existing, chỉ thêm/update — không cần rollback logic cho Fix 3+4.
```

### Rollback Fix 5 (Web UI)
